# M6 扩展功能需求 — AI Agent 深度增强

> 本文档基于 cognee 现有代码的深度分析 + AI Agent 领域最佳实践，提出尚未在 M1-M5 中覆盖的功能模块。

## 6.1 多语言内容翻译

### 功能描述
系统内置了完整的多语言翻译 Pipeline Task，能够自动检测文档语言并翻译为目标语言，使知识库具备跨语言检索和理解能力。

### 核心能力
| 能力 | 说明 |
|------|------|
| **语言自动检测** | 使用 `detect_language_async` 对每个 chunk 进行语言识别，返回语言代码 + 置信度 |
| **多翻译引擎** | 支持 LLM 翻译、Google Translate、Azure Translator 三种 Provider |
| **原文保留** | 翻译后原文保留在 `TranslatedContent` 中，支持双语对照 |
| **置信度过滤** | 可配置最低置信度阈值，低置信度的检测结果跳过翻译 |
| **Pipeline 集成** | 作为标准 Task 嵌入 cognify pipeline，与其他 Task 串联 |

### 前端需求
- 知识库语言统计仪表盘（各语言占比）
- 翻译配置（目标语言、翻译引擎选择、置信度阈值）
- 翻译进度和质量预览

## 6.2 Web 内容抓取与定时爬取

### 功能描述
系统内置了 Web 爬虫任务，支持单次抓取和 Cron 定时调度抓取，将网页内容结构化后存入知识图谱。

### 核心能力
| 能力 | 说明 |
|------|------|
| **单页抓取** | 传入 URL，抓取页面内容并解析为结构化数据 |
| **批量抓取** | 支持 URL 列表批量抓取 |
| **Cron 定时抓取** | 使用 APScheduler 实现 Cron 表达式定时调度 |
| **数据建模** | WebPage、WebSite、ScrapingJob 三个 DataPoint 模型 |
| **多爬虫引擎** | 默认 BeautifulSoup 爬虫 + Tavily API 爬虫 |
| **增量更新** | 基于 URL 哈希判断页面是否已存在，支持更新 |

### 前端需求
- 爬取任务管理（创建/编辑/删除/暂停）
- Cron 表达式配置界面
- 爬取历史和状态监控
- 抓取内容预览

## 6.3 反馈驱动的答案改进（Feedback Loop Pipeline）

### 功能描述
系统内置了一套完整的反馈驱动改进流水线，能够自动从负面反馈中提取改进线索、生成改进版答案、并将改进结果关联回知识图谱。

### Pipeline 步骤
| 步骤 | Task | 说明 |
|------|------|------|
| 1 | `extract_feedback_interactions` | 从图谱中提取负面反馈节点及其关联的用户交互 |
| 2 | `generate_improved_answers` | 使用 LLM 基于原始问题、上下文和负面反馈生成改进版答案 |
| 3 | `link_enrichments_to_feedback` | 将改进结果（FeedbackEnrichment）通过边关联回原始反馈节点 |

### 核心特点
- **自动化闭环**: 从用户反馈 → 问题定位 → 答案改进 → 图谱增强，全自动完成
- **负面反馈聚焦**: 自动过滤出 sentiment=negative 或 score<0 的反馈
- **可追溯改进**: 改进结果通过 `enriches` 边关联到原始反馈和交互节点

### 前端需求
- 反馈改进任务的触发和监控
- 改进前/后答案对比展示
- 改进建议的人工审核和确认

## 6.4 代码图谱分析

### 功能描述
系统支持从代码文件中提取结构化知识图谱，将代码实体（类、函数、模块等）及其依赖关系建模为图谱节点和边。

### 核心能力
| 能力 | 说明 |
|------|------|
| **代码图谱提取** | `extract_graph_from_code` 从代码 chunk 中使用 LLM 提取知识图谱 |
| **代码摘要** | `summarize_code` 对代码节点生成自然语言摘要 |
| **自定义图模型** | 支持传入自定义 Pydantic Graph Model 控制提取的实体类型 |

### 前端需求
- 代码仓库导入（Git URL 或本地路径）
- 代码图谱可视化（类/函数/模块关系图）
- 代码实体搜索

## 6.5 数据库 Schema 逆向工程

### 功能描述
系统支持从关系型数据库中逆向提取 Schema 元数据（表、列、关系），构建数据库知识图谱，便于数据资产管理和自然语言查询。

### 核心能力
| 能力 | 说明 |
|------|------|
| **Schema 提取** | `ingest_database_schema` 从数据库中提取表结构、列信息、外键关系 |
| **样本数据** | 可选提取每张表的 N 行样本数据 |
| **DataPoint 建模** | DatabaseSchema、SchemaTable、SchemaRelationship 三个模型 |

### 前端需求
- 数据源连接管理（添加数据库连接字符串）
- Schema 可视化（ER 图）
- 自然语言查询数据库

## 6.6 分布式执行引擎

### 功能描述
系统通过 `distributed/` 模块支持基于 Modal 平台的分布式 Pipeline 执行，将计算密集型任务分发到云端 Worker 执行。

### 核心能力
| 能力 | 说明 |
|------|------|
| **Modal 集成** | 使用 Modal App 进行云端函数调度 |
| **任务队列** | `add_nodes_and_edges_queue`、`add_data_points_queue` 两个队列 |
| **Worker 分离** | 独立的 Worker 容器处理图谱写入操作 |
| **Docker 部署** | 提供独立的 Dockerfile 用于 Worker 部署 |

### 前端需求
- 分布式任务监控面板
- Worker 节点状态查看
- 任务队列深度和处理速度指标

## 6.7 多数据源 Loader 生态

### 功能描述
系统通过 `LoaderInterface` 抽象接口和 `LoaderEngine` 统一管理，支持 30+ 种数据源的加载。

### 已实现的 Loader
| Loader | 支持格式 |
|--------|----------|
| **TextLoader** | .txt, .md, .json 等纯文本 |
| **CsvLoader** | .csv 结构化数据 |
| **PyPdfLoader** | .pdf（基础 PDF 解析） |
| **AdvancedPdfLoader** | .pdf（高级 PDF 解析，支持表格/图片提取） |
| **UnstructuredLoader** | 多格式文档（通过 Unstructured.io） |
| **BeautifulSoupLoader** | HTML 网页内容 |
| **AudioLoader** | 音频文件（通过 Speech-to-Text 转录） |
| **ImageLoader** | 图片文件（通过 OCR/Vision 提取文本） |

### 前端需求
- 支持的文件格式列表展示
- 文件上传时自动识别并选择 Loader
- 各 Loader 的高级配置选项

## 6.8 级联图谱提取（Cascade Extract）

### 功能描述
标准的单次 LLM 提取可能遗漏复杂文档中的深层实体关系。级联提取通过多轮迭代方式逐步深入挖掘文档中的知识。

### 与标准提取的对比
| 维度 | 标准提取 | 级联提取 |
|------|---------|---------|
| **轮次** | 单次 LLM 调用 | 多轮迭代 |
| **深度** | 表层实体和关系 | 多层嵌套关系 |
| **成本** | 低 | 较高（多次 LLM 调用） |
| **适用场景** | 短文本、简单关系 | 长文档、复杂领域知识 |

### 前端需求
- Cognify 时选择提取模式（标准/级联）
- 每轮提取结果的可视化展示

## 6.9 Ontology 驱动的图谱对齐

### 功能描述
系统支持使用 OWL 本体文件约束图谱提取过程，确保提取出的实体和关系符合预定义的领域模型。同时支持在图谱构建后对已有节点进行本体对齐验证。

### 核心能力
| 能力 | 说明 |
|------|------|
| **本体约束提取** | Cognify 时传入 ontology_id，实体类型和关系类型受 OWL 约束 |
| **OntologyResolver** | 图谱提取后使用 Resolver 进行子图匹配和验证 |
| **环境级配置** | 通过 `GRAPH_MODEL_FILE` + `GRAPH_MODEL_NAME` 全局指定图模型 |
| **自定义 Graph Model** | 用户可通过 Pydantic 类定义自定义图模型替代默认 KnowledgeGraph |

### 前端需求
- OWL 文件管理（上传/查看/删除）— 与 M2 本体管理集成
- Cognify 高级配置中选择本体文件和图模型
- 图谱对齐报告（哪些节点不符合本体约束）

## 6.10 智能搜索路由（Feeling Lucky）

### 功能描述
系统已有 `FEELING_LUCKY` SearchType，会智能选择最合适的搜索策略。建议扩展为更完善的「智能搜索路由」功能。

### 建议增强
| 增强点 | 说明 |
|--------|------|
| **意图识别** | 使用 LLM 或轻量模型分析查询意图（事实查询/关系查询/代码查询/总结） |
| **策略推荐** | 基于意图自动选择最优 SearchType（GRAPH_COMPLETION / CHUNKS / CODING_RULES 等） |
| **多策略融合** | 对同一查询并行执行多种搜索，融合结果后返回最优答案 |
| **用户偏好学习** | 基于用户历史搜索行为和反馈，个性化调整搜索策略偏好 |

### 前端需求
- 搜索结果来源标注（标注使用了哪种搜索策略）
- 搜索策略手动切换（高级用户可覆盖自动选择）

## 6.11 知识图谱冲突检测与合并

### 功能描述
当不同文档或不同时间的 Cognify 产生矛盾信息时（如同一实体的不同属性值），需要自动检测和解决冲突。

### 核心功能
| 功能 | 说明 |
|------|------|
| **冲突检测** | 同一实体的不同来源属性值比较，标记矛盾 |
| **冲突展示** | 高亮显示冲突节点，展示各来源的不同说法 |
| **自动合并策略** | 按时间最新/来源权重/LLM 仲裁 三种策略 |
| **人工裁决** | 提供 UI 让用户手动选择保留哪个版本 |

### 前端需求
- 冲突列表面板（按严重程度排序）
- 冲突详情对比视图
- 一键合并 / 人工选择 操作

## 6.12 Agent 记忆衰减与遗忘机制

### 功能描述
模拟人类记忆的遗忘曲线，让长期不被访问的知识自动降低优先级，确保 Agent 的"记忆"始终保持高相关性和新鲜度。

### 核心功能
| 功能 | 说明 |
|------|------|
| **访问计数** | 每次搜索命中的节点增加访问计数 |
| **时间衰减** | 基于 Ebbinghaus 遗忘曲线对节点权重施加时间衰减因子 |
| **冷热分层** | 热数据保持高优先级，冷数据降低排序权重 |
| **主动遗忘** | 用户可标记某些知识为"已过时"，加速衰减 |
| **复习强化** | 被反复访问的知识获得"间隔重复"加成，类似 SRS |

### 与现有系统的关系
- 可复用 `apply_feedback_weight()` 机制动态调整节点权重
- 可结合 Temporal Awareness 的时间维度能力

### 前端需求
- 知识"健康度"仪表盘（活跃/沉睡/过时知识占比）
- 知识保鲜设置（衰减速率、最低保留权重）

## 6.13 LLM 结构化输出能力

### 功能描述
cognee 的 `LLMGateway.acreate_structured_output` 支持将 LLM 输出强制约束为 Pydantic 模型结构，是所有图谱提取、规则提取、反馈分析的基础能力。

### 核心特点
| 特点 | 说明 |
|------|------|
| **多框架支持** | BAML 或 LiteLLM+Instructor，通过 `STRUCTURED_OUTPUT_FRAMEWORK` 配置 |
| **类型安全** | 输出直接映射为 Pydantic BaseModel 实例 |
| **错误重试** | 结构不匹配时自动重试 |
| **自定义模型** | 用户可定义任意 Pydantic 类作为输出结构 |

### 前端需求（面向开发者）
- 结构化输出测试 Playground（定义 Pydantic 模型 + 输入文本 → 查看提取结果）
- 框架选择配置

## 6.14 多租户数据隔离与权限

### 功能描述
系统通过 `ENABLE_BACKEND_ACCESS_CONTROL` + ACL (Access Control List) 模型实现多租户场景下的数据隔离。

### 现有能力
| 能力 | 说明 |
|------|------|
| **User / Principal** | 用户和权限主体模型 |
| **ACL** | 访问控制列表，绑定 (Principal, Dataset, Permission) 三元组 |
| **Dataset 级隔离** | 搜索时自动按用户权限过滤 Dataset |
| **默认权限** | `UserDefaultPermissions` 模型定义新用户的默认权限 |
| **权限授予** | `give_permission_on_dataset` 方法实现权限分配 |

### 建议增强
| 增强 | 说明 |
|------|------|
| **组织/团队** | 在 User 之上增加 Organization / Team 层级 |
| **角色模板** | 预定义角色（Admin / Editor / Viewer）绑定权限集 |
| **权限继承** | 组织级权限自动继承到成员 |
| **邀请机制** | 通过邮件/链接邀请用户加入团队 |

### 前端需求
- 用户管理页面（列表/邀请/禁用）
- 权限矩阵管理（用户 × Dataset × 权限类型）
- 组织/团队管理
